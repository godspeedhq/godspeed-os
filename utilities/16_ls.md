# Utility: `ls` — list a directory

**Status:** **Built + QEMU-verified** (`osdev test files` 11/11) — a shell built-in over
the `fs` LIST_DIR API, on hierarchical GSFS (`docs/persistence.md`). Read-only. Trails
`CLAUDE.md`; does not amend it. Shared addressing/current-location rules live in `17_cd.md`
and `docs/drives.md` §4.1.

---

## 1. What it is

`ls` lists the entries of a directory — each name, whether it is a file or a directory,
and (for files) the size. With no argument it lists the **current directory** (`20_cd.md`);
with a path it lists that directory. `ls` is read-only: it never changes the disk.

`ls` is kept (not renamed to `list`) because it is universal and unambiguous — one of the
three short verbs we keep (`ls` / `cd` / `mkdir`), unlike the cryptic ones we dropped.

## 2. Usage

```
ls 0.1.0 — list a directory (records when piped)

usage:
  ls                       list the current directory
  ls <path>                list the directory at <path>
  ls [path] | <verb>       piped: emits records (name/type/size)
  ls | select … / sort …   project / order the listing
  ls version               print the version
  ls help                  print this message

<path> = [index:]label/path | /abs | rel   (see docs/drives.md §4.1)
```

Example:

```
gs> ls /projects
  NAME            TYPE   SIZE
  notes.txt       file   18 B
  drafts          dir    —
```

## 2a. As a record producer (typed pipes)

`ls` is a **record producer** (`docs/records.md`, `utilities/31_records.md`): bare it prints
the text listing above, but **in a pipe** it emits a typed **table** with columns
**`name` / `type` / `size`** (`type` is `file`/`dir`; `size` is the byte count for files and
empty for directories). So the structured verbs operate on real fields instead of re-parsing
text:

```
ls | where type=file               only files
ls /docs | where size>0            non-empty files
ls | select name size              keep two columns
ls | where type=file | sort reverse size   biggest files first
ls | to json                       the listing as JSON
```

This is the structured replacement for the old plan to bolt name/size sorting onto `ls` itself
(§5) — sorting and filtering are the pipe's job (`where`/`select`/`sort <col>`), not flags on
`ls`. A *text* filter (`match`/`count`/`first`/`last`) on `ls` output is a loud, guided error
(it is records, not text — use `where`, or `to json` to get text first).

## 3. Addressing & current location

`<path>` is the standard file address: a `[index:]label/path` on any present drive, a
`/absolute` path on the current drive, or a `relative` path from the current directory.
The current location is one drive+directory pointer moved by `cd` (`20_cd.md`).

## 4. Implementation

Read-only, so `ls` is a **shell built-in** that sends `ListDir` to `fs` (op 14) over a
narrow `ipc_send=["fs"]` cap and formats the reply. `fs` enforces; the shell's authority is
not widened beyond reading (`0_conventions.md` §2). Two code paths share that one `fs` call:
`cmd_ls` formats the text listing (bare `ls`), and `build_ls_table` parses the same reply into
the `name`/`type`/`size` table for the record pipe (`is_record_producer` routes a piped `ls` to
it before the text producers are consulted).
